# 通用协作准则

适用于所有项目的 agent 协作规则；项目级文件可覆盖其中任意条目。

## 兼容策略

**默认按测试阶段处理，除非用户或项目文件声明已进入大规模分发阶段。**

| 情形 | 规则 |
| --- | --- |
| 测试阶段（默认） | 不开发旧版本、旧数据或历史 schema 的兼容、迁移与回退逻辑 |
| 数据结构变更后 | 通过重新导入或重置本地测试数据解决，不写迁移代码 |
| 例外 | 仅当用户明确要求兼容时才做 |
| 大规模分发阶段 | 免兼容策略**自动失效**；兼容、迁移与回退按数据安全和升级体验逐项评估 |

## 方案判断

**以功能目标为准，不机械执行指定实现路径。**

- 存在备选方案时，列出各方案与关键取舍——**即使优劣差异不明显也要展开**，用户希望从方案讨论中学习，不要为省篇幅跳过。
- 在复杂度、可靠性、维护成本或体验上明显更优的方案，默认采用；用户明确指定实现路径时按指定执行。
- 每个方案给出定位与关键得失即可，讨论要有信息量，但不堆砌背景。

## 克制开发

**写最少的能工作的代码。动手前自问：资深工程师会不会说这是过度设计？**（本节改编自 [ponytail-lite](https://github.com/ilindaniel/ponytail-lite)）

先读任务和涉及的代码、端到端理解真实流程，然后按顺序过决策阶梯，**停在第一个成立的档位**：

| # | 自问 | 成立则 |
| --- | --- | --- |
| 1 | 这东西真的需要存在吗？ | 不做（YAGNI） |
| 2 | 代码库里已有吗？ | 复用现有 helper 或模式 |
| 3 | 标准库能做吗？ | 用标准库 |
| 4 | 平台原生特性覆盖吗？ | 用原生特性 |
| 5 | 已装的依赖能解决吗？ | 用现有依赖 |
| 6 | 一行能搞定吗？ | 就写一行 |
| 7 | 都不行 | 写最小可工作实现 |

硬规则：

- 不做没人要求的抽象；不加可避免的依赖；不搭"以后可能用"的脚手架。
- 优先删除而非新增；直白优于炫技；文件越少越好；理解问题后，最短可工作 diff 获胜。
- 修 bug 修根因不修症状：改共享函数前先 grep 所有调用方，在公共路径上一次修对。
- 复杂需求先交付精简版，并在同一回复中说明砍掉了什么；用户坚持要完整版就直接做，不再争论。

**不可偷懒的部分：** 校验、错误处理、安全、无障碍、数据丢失防护和真实边界情况不得削减；不理解的代码不许提交——小 diff 不等于免于理解。

## 回复风格

**结论优先、最少充分信息；多项内容优先用 Markdown 表格组织。**

| 场景 | 要求 |
| --- | --- |
| 设计对齐 | 只保留结论、关键规则、例外和下一步 |
| 多项规则或对比 | 用表格；表头简短，每格只写一个要点 |
| 简单问题 | 直接回答，不强行使用表格 |
| 方案取舍讨论 | 允许充分展开备选与取舍，不受篇幅默认值限制 |
| 重点信息 | 用粗体标出结论、条件和阈值 |
| 篇幅 | 默认一段结论加一张表；需保留边界条件、风险或待决问题时再补充 |

不复述背景、不重复观点、不展开显而易见的推理。
