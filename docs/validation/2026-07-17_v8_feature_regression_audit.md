# V8 功能回归全面审计报告

审计日期：2026-07-17  
审计对象：XL Translator V8.0.0  
当前版本：`v8.0.0` / `e2660c3`  
对照基线：`v7.4.2` / `14c2e55`

## 1. 结论

本次 V8 Tauri 升级存在系统性用户功能回归，不是少数控件遗漏。

以“用户可以独立操作或感知的一项能力”为统计单位，本次静态审计确认 **63 个能力点发生回归**：

| 回归类型 | 数量 | 含义 |
| --- | ---: | --- |
| 入口或结果呈现缺失 | 48 | 底层数据或核心逻辑大多仍在，但 V8 UI 无法操作或查看 |
| 行为缩水、错误或实际失效 | 15 | UI 看似存在但功能不完整，或在特定流程中直接失败 |
| 合计 | **63** | 不重复按每个页面或每种语言机械计数 |

其中至少 7 项属于会阻断正常使用、破坏数据保护语义或造成任务状态错误的严重问题：

1. 任意多语言互译被固定语言列表截断。
2. 固定 TM 词条失去编辑、删除保护。
3. `.xls` 兼容模式 fallback 无法启用。
4. PNG/JPG/WebP/BMP/TIFF 图片无法进入 PDF/图片任务清单。
5. PDF 审核启用前不再检查审核模型配置。
6. PDF 运行期模型可用性状态不能按旧流程持久化。
7. 多任务并行时单一 `state.running` 可能导致任务状态和 SSE 事件串线。

本轮没有发现用户原有语言配置、模型角色配置、Prompt 或 TM 数据被主动删除。主要问题是成熟 Qt UI 被移除后，相应功能没有完整迁移到 Tauri UI。

## 2. 审计范围与方法

本报告对照以下内容：

- `v7.4.2` 的 `native_app/` Qt 界面、页面生命周期和回归测试。
- V8 的 `ui/src/main.ts`、`ui/src/api-client.ts`、`api/app.py` 和 `api/task_manager.py`。
- 当前 `settings.py`、`core/language_registry.py`、`core/tm_manager.py` 等底层能力。
- Route B 实施文档和最终验收记录。
- V8 UI 实际调用的 API 与后端已经实现的 API 路由。

关键代码规模变化：

| 范围 | 新增 | 删除 |
| --- | ---: | ---: |
| `native_app/` → `ui/`/Tauri 相关差异 | 2,929 行 | 14,035 行 |
| 当前主要页面逻辑 `ui/src/main.ts` | 约 933 行 | — |
| 被删除的 5 组 Qt UI 回归测试 | — | 3,466 行 |

当前 `ui/package.json` 只有 TypeScript 检查和 Vite 构建，没有 Vitest、Playwright、Cypress 或其他 UI 行为测试。现有“全量测试通过”只能证明后端和构建链路可用，不能证明 V7.4 用户功能等价。

## 3. 多语言与自定义语言回归

### 3.1 数量变化

V7.4.2 通过 `core/language_registry.py` 动态提供：

- 58 种常规目标语言。
- 1 种可选内置语言（中文）。
- 合计 59 种内置源/目标语言。
- 用户已保存的自定义语言。
- 语言代码、名称和别名搜索。
- 最近使用语言排序。

V8 在 `ui/src/main.ts` 中硬编码为：

- 目标语言 5 种：中文、英文、法文、阿拉伯语、日语。
- 源语言 4 种：中文、英文、法文、阿拉伯语。

因此：

- 目标语言隐藏了 54 种内置语言。
- 源语言隐藏了 55 种内置语言。
- PDF 目标语言同样隐藏了 54 种内置语言。
- 自定义语言完全不显示。
- 任意语言对互译被限制为固定下拉框能组合出的少量语言对。

### 3.2 底层状态

以下底层能力仍然存在：

- `CUSTOM_TARGET_LANG_PREFIX` 和自定义语言编码。
- 自定义语言新增、描述更新、删除和校验函数。
- `custom_target_langs`、`recent_target_langs` 设置字段。
- 内置/可选/自定义语言排序。
- 中文语言名称别名和语言代码搜索。
- 自定义语言说明注入模型 Prompt。

结论：语言能力主要是 UI 可达性回归，恢复时不需要重写翻译引擎。

## 4. 记忆库/知识库回归

V8 使用全局翻译设置拼接当前 TM 语言对：

```ts
function tmLangPair(): string {
  return `${source_lang}-${target_lang}`;
}
```

这意味着记忆库不再拥有独立的源语言和目标语言选择器。由于全局语言下拉框本身已经被截断，原有其他语言对和自定义语言的 TM 数据虽然还在数据库里，但无法在 V8 中正常浏览、搜索、清洗或导出。

其他确认回归包括：

- 只请求第一页，默认最多显示 50 条，无分页控件。
- 词长上限设置消失。
- 批量固定、全部解锁和批量删除消失。
- CSV 导入、CSV 导出消失。
- 导入重复项策略固定为 `skip`，不能选择覆盖。
- 固定词条仍显示编辑和删除按钮，失去旧版保护语义。
- 深度清洗只保留差异确认模式。
- 清洗后自动固定被硬编码为 `false`。
- 分语言对补充 Prompt、完整 Prompt 覆盖和实际发送预览消失。
- 清洗进度、停止/取消能力消失。

TM CRUD、清洗、导入导出和批量固定 API 仍部分存在，数据库结构也未被本次迁移删除。

## 5. 专业领域与模型配置回归

V8 将旧版多角色模型配置压缩为一套全局引擎表单。以下设置仍存在于 `settings.py`，但没有对应 UI：

- 专业领域预设选择。
- 预设 Prompt 查看和编辑。
- 预设 Prompt 覆盖和自定义领域 Prompt。
- 翻译、清洗、图片、PDF 审核模型角色。
- 模型角色的独立配置和跟随关系。
- 清洗/图片角色独立服务商、API Key、Base URL 和模型。
- PDF 审核模型独立服务商、API Key、Base URL 和模型。
- 图片模型连接测试。
- PDF 审核模型连接测试。
- 每个有效模型独立保存的批次大小和并发数。

另外两项行为发生缩水：

- 切换服务商时不会恢复该服务商独立保存的模型和 Base URL。
- “获取模型”只用 toast 显示前 6 个模型名称，不能在模型下拉框中选择。

后端已有模型角色、吞吐量、图片连接和 PDF 审核连接 API，但 V8 UI 没有调用。

## 6. Excel 回归

基础扫描、文件选择和任务启动仍然存在，但以下行为发生回归：

- `.xls` 文件在本地 Excel 自动化不可用时，不再询问是否使用兼容模式。
- V8 启动任务时没有传入 `allow_xls_fallback: true` 的路径。
- 自定义输出目录不再实时校验，也不显示“将创建”“可用”或错误状态。
- `.xls` 数量和总工作表数等扫描概况被简化。
- 完成后的输出目录、文件状态和错误原因进入统一结果呈现回归。

## 7. Word 回归

以下旧版设置仍存在于后端，但 V8 页面没有对应控件：

- 原有高亮处理策略。
- 本地 Word/LibreOffice 自动编号预处理。
- 长段拆分阈值。
- 严格重试次数。

此外：

- 扫描表格不再展示每个 Word 文件的表格数量。
- 后端仍发送 `word_recovery` 事件，但 V8 不处理和呈现。

V8 目前只保留段落上限、字符上限、复核开关和三种复核颜色等部分设置。

## 8. PDF 与图片翻译回归

### 8.1 图片输入实际不可用

V8 虽然显示“图片翻译”开关，但扫描请求被硬编码为：

```json
{"include_images": false}
```

文件选择器也只允许选择 PDF。因此 V7.4.2 支持的 PNG、JPG、JPEG、WebP、BMP、TIFF 图片无法进入任务清单。

### 8.2 审核、恢复和结果工作流

以下能力消失或失效：

- 开启 PDF 审核前的审核模型配置检查。
- “停止提交新页”后继续翻译。
- 继续翻译 API 和界面操作。
- 页级重试的“重试中/已恢复/占位页”实时指标。
- PDF 审核的“审核中/已通过/需复核”实时指标。
- 中止后的输出目录、报告、manifest 和页面素材路径。
- 中止后逐文件完成状态。
- 运行期图片/审核模型可用性状态的旧版持久化流程。

## 9. 任务生命周期与结果呈现回归

### 9.1 结果页

Excel、Word、PDF 共同丢失：

- 耗时、TM 命中、API 次数、成功/失败数等结果 KPI。
- 输出目录显示。
- 每个文件的成功/失败状态。
- 每个文件的错误原因。
- Word/PDF 问题定位表。
- PDF 占位页、审核失败和恢复详情。
- 独立的异常和中止结果工作区。

V8 只保留最后 10 条日志，并使用 toast 提示任务完成，用户难以定位产物和失败原因。

### 9.2 任务锁与并发

后端 `TranslationTaskManager` 仍支持资源锁和互不冲突的并行任务，但前端只有一个全局 `state.running`：

- 启动第二个任务会覆盖第一个任务的界面状态。
- 两个异步 SSE 监听都通过同一个 `handleTaskEvent()` 修改当前 `state.running`。
- 事件可能被写入错误任务的状态对象。
- UI 没有调用 `/api/tasks/locks/current`，不能提前锁定冲突页面。
- 冲突通常只能在任务启动后通过服务端 409 错误发现。

### 9.3 页面状态恢复

V8 会写入 `last_excel_source_folder`、`last_word_source_folder` 和 `last_pdf_source_folder`，但初始化后没有把它们恢复到 `state.sourcePaths`，所以上次使用路径不能自动回填。

## 13. 验收失效原因

Route B 实施文档明确要求：

> 用户可感知功能与 V7.4 等价（逐控件对照）

Phase 2 还要求每页：

1. 与 Qt 版逐控件功能对照。
2. 与原型进行双主题视觉对照。
3. 验证 running 状态 SSE 日志和进度。
4. 完成中文 tooltip 文案迁移。

最终验收记录只确认四个视图、模型配置抽屉和基础任务控件存在，并以可访问性树和 sidecar/CORS 验证代替截图，没有提供逐控件对照矩阵，也没有证明每个旧控件对应的新入口已经实现。

与此同时，原有 Qt UI 行为测试被删除，而新 UI 没有同等的自动化行为测试。这使得“页面存在”和“项目可以构建”被错误地当成了“功能等价”。

## 14. 恢复优先级

### P0：立即恢复可用性和数据保护语义

1. 恢复完整语言注册、自定义语言和任意语言对。
2. 恢复 TM 独立语言对选择，确保原多语言词库可访问。
3. 修复 PDF/图片扫描的 `include_images` 和文件过滤器。
4. 恢复 `.xls` 兼容 fallback。
5. 将单一 `state.running` 改为按任务/页面隔离的状态模型。
6. 恢复固定 TM 词条保护。

### P1：恢复核心专业工作流

1. 恢复专业领域预设和 Prompt 编辑。
2. 恢复翻译、清洗、图片、PDF 审核模型角色及跟随关系。
3. 恢复 PDF 停止提交/继续翻译和审核前置检查。
4. 恢复 TM 清洗模式、Prompt、自动固定、分页和批量操作。

### P2：恢复结果可追踪性

1. 恢复输出目录、逐文件状态和错误原因。
2. 恢复 Word/PDF 问题定位表和恢复/审核指标。
3. 恢复完整日志、异常和中止工作区。
4. 恢复上次源路径和输出目录校验。

### P3：恢复辅助能力并防止再次回归

1. 恢复更新链接、忽略/恢复和自动通知。
