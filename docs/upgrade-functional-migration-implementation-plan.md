## 2. 已冻结的范围边界

- 新版建立全新的数据基线；不读取、迁移、修复或删除本次新基线以前的设置、Key、TM、模型导出或自定义语言数据。
- 新版只发布 macOS，最低支持 macOS 12.0 Monterey；Windows 继续使用旧版本，不进入新版实现和验收。
- macOS 发布必须提供 Apple Silicon `arm64` 和 Intel `x86_64` 两个原生 DMG。
- 用户界面和下载说明可显示“Intel x64”；构建配置、CI、Mach-O 扫描和验收记录统一使用技术架构名 `x86_64`。
- 新版应用数据目录固定为 `~/Library/Application Support/Translator`；旧版 `~/.xl_translator` 及其他旧路径完全不读取、不复制、不修改。
- 真实服务 Key 连通性和真实 API 翻译验收继续暂缓；本地实现阶段使用隔离 Mock API 和固定样例完成可重复验收。
- 不因迁移到 Tauri 而删除已在最终决策文档中确认的语言、TM、模型、Excel、Word、PDF/图片、调度、结果、帮助和诊断能力。
- 自定义语言只作为目标语言；自动识别按文件预检最多返回两个实际源语言；TM 不承担语言识别职责。

## 3. 执行模型

### 3.1 四条并行线

每个阶段都提前拆成四条执行线。三条由子 Agent 并行执行，第四条由本体负责跨线集成和阶段门禁。

| 执行线 | 默认负责人 | 固定职责 | 交付边界 |
| --- | --- | --- | --- |
| L1 核心与契约 | 子 Agent 1 | Python core、API、数据结构、任务语义、服务端逻辑 | 只提交后端/数据/API 所属文件及契约样例；不直接改 UI 页面 |
| L2 Tauri 与 UI | 子 Agent 2 | vanilla TypeScript、CSS、Tauri Rust、原生权限和打包入口 | 只提交 UI/`src-tauri/`/平台配置所属文件；不改变核心业务规则 |
| L3 测试与兼容 | 子 Agent 3 | 单元/集成/动态测试、隔离夹具、Mock API、macOS 12/发布验证、文档证据 | 只提交测试、验证脚本、验收记录和必要的发布文档 |
| L4 主线集成与门禁 | 本体 | 合并三条线、解决契约冲突、执行跨模块复核、运行阶段验收并决定是否进入下一阶段 | 可以修改任意文件，但必须保留每条线的变更来源和验收证据 |

### 3.2 并行协作规则

1. 每条线先读取本阶段的入口条件和最终决策编号，再开始修改。
2. 共享契约文件先由 L1 提交草案，L2/L3 通过示例和测试消费；出现冲突时暂停相关线，由 L4 统一裁决。
3. 三条子 Agent 不直接修改同一业务文件；需要跨线修改时先提交接口或测试夹具，再由 L4 合并。
4. 阶段内允许四条线并行；阶段之间严格串行。未通过阶段门时，下一阶段不得开始编码。
5. 每条线结束必须提交：变更清单、验证命令、验证结果、未覆盖项和潜在风险。

### 3.3 每阶段启动模板

阶段开始时，L4 必须先建立本阶段的三个子 Agent 任务，并同步建立本体 L4 的阶段开工记录，再开始本体集成工作。四者属于同一个阶段波次，允许在阶段内部并行；三个任务使用固定命名，便于日志、提交和验收记录追踪：

| 任务名 | 对应执行线 | 必须先产出的内容 |
| --- | --- | --- |
| `phase-N-core` | L1 核心与契约 | 契约草案、核心实现清单、接口样例 |
| `phase-N-ui` | L2 Tauri 与 UI | 页面/壳层实现清单、交互状态清单 |
| `phase-N-test` | L3 测试与兼容 | 隔离夹具、动态测试矩阵、验收脚本 |

其中 `N` 为两位阶段编号（例如 `phase-02-core`）。L4 在阶段记录中登记三个任务的启动时间、完成时间、提交 SHA、验证命令和结果；任一子 Agent 未完成或未提交验证证据，L4 不得宣布该阶段通过。子 Agent 只在当前阶段工作，不得提前修改下一阶段的业务范围。

每个阶段的最小开工记录必须包含：上一阶段验收文件和状态、当前阶段决策编号、三个子 Agent 任务名、L4 集成责任、共享契约文件、隔离产物目录以及本阶段的阻断条件。没有这份记录，不得把并行线的代码或测试结果计入阶段验收。

### 3.3.1 证据与运行环境约束

- 每条线的可追溯证据必须是提交 SHA，或在尚未提交时可复现的补丁摘要（文件清单与 SHA-256）；“当前工作树有改动”本身不能作为阶段通过证据。历史 Phase 0–2 的补录证据不追溯改写；Phase 3 起严格执行本规则。
- 所有 Python 实现和测试统一使用 `./.venv/bin/python3`；不得混用系统 Python。涉及 `settings.json`、`keys.json`、TM、临时目录或用户目录时，必须在导入业务模块前隔离 `HOME`、`USERPROFILE`、`TEMP`、`TMP` 或 `TRANSLATOR_APP_DATA_DIR`。
- `quality_gate.ps1` 是必经检查，但不替代本阶段所需的 UI、Rust、动态或发布验证；验收记录必须逐项列出实际执行的命令与结果，不能把未运行项概称为“通过”。

### 3.4 每阶段通用交付门

每个阶段必须同时满足以下条件，才算通过：

1. L1、L2、L3、L4 四条线均有完成记录。
2. 已执行 `powershell -ExecutionPolicy Bypass -File ./quality_gate.ps1`。
3. 至少执行一个与本阶段直接相关的动态测试；产物统一放在 `.runtime/self-tests/<phase-slug>/`。
4. 涉及 UI/Tauri 时，额外执行 `ui/` 的 `npm run check && npm run build`、`src-tauri/` 的 `cargo test && cargo check`，并用隔离 `TRANSLATOR_APP_DATA_DIR` 启动当前源码验证。
5. UI/Tauri 测试收尾时关闭旧的 `Translator`、`tauri dev` 和 `api.launcher` 进程，记录新进程 PID、启动时间、启动路径和 sidecar `/health`。
6. 阶段验收记录包含通过项、失败项、未执行项、阻塞原因和风险范围。
7. 任何 P0/P1 功能回归、数据损坏、密钥泄露、错误语言对写入或 macOS 12 构建门失败，均阻断阶段通过。

### 3.5 阶段汇合与串行放行

每个阶段均按以下顺序运行；“三线都完成”不等于“阶段已通过”：

1. **L4 开工单**：确认上一阶段验收记录为 `passed`，登记本阶段决策编号、入口条件、共享契约、文件边界和三个子 Agent 任务名。
2. **三线并行**：同时启动 `phase-N-core`、`phase-N-ui` 和 `phase-N-test`。三条线只处理当前阶段，不提前修改后续阶段范围。
3. **契约汇合**：L1 先提供可运行接口样例，L2/L3 基于该样例消费；共享契约发生变化时，L4 记录影响范围并要求受影响线重验。
4. **四线交付**：L1/L2/L3 分别提交变更清单、验证命令、产物路径、未覆盖项和风险；L4 完成跨线集成、冲突处理和阶段专属动态测试。
5. **验收记录**：L4 将结果写入 `docs/validation/<YYYY-MM-DD>_<phase-slug>_acceptance.md`，状态只能是 `passed` 或 `blocked-by-gate`。
6. **放行规则**：只有状态为 `passed` 且阻断项为空，才允许创建下一阶段开工单；否则留在当前阶段修复并重新执行第 3–5 步。

```text
上一阶段 passed
        |
        v
L4 开工单 -> L1 core
          -> L2 ui       } 同阶段并行
          -> L3 test
        |
        v
L4 汇合 + 质量门 + 动态测试 + 验收记录
        |
   passed ? ---- 否 ----> 当前阶段修复并重验
        |
       是
        v
下一阶段开工单
```

本体 L4 负责接口仲裁、跨线集成、验收和放行，不替代三个子 Agent 的专责交付；三个子 Agent 也不得自行宣布阶段通过。

## 5. Phase 0：基础契约与兼容性基线

### 5.1 目标

先建立所有后续模块共用的接口、全新数据基线、隔离测试容器和 macOS 12 发布硬约束。此阶段不恢复旧数据迁移，不开始大规模页面功能开发。

### 5.2 四条并行线

**L1 核心与契约（子 Agent 1）**

- 定义语言目录、实际源语言代码、目标语言代码、自定义语言内部代码和任务语言快照的数据结构。
- 定义 TM 条目、可信状态、清洗候选、模型角色、任务资源、结果终态和诊断记录的当前 schema。
- 固化 API 错误协议、分页/批量协议、SSE 事件协议和任务 ID 语义。
- 明确新应用数据目录和 schema 版本；删除设计上对旧版数据目录的读取入口。

**L2 Tauri 与 UI（子 Agent 2）**

- 核对 Tauri 窗口、sidecar、随机 loopback、token 握手、单实例和退出清理的基础接口。
- 建立 `safari15.1` 构建基线，清理或为 `color-mix()` 等 Safari 15 不支持的 CSS 提供静态回退。
- 设计 macOS 12/13+ Apple Events 权限提示文案和 `NSAppleEventsUsageDescription`、自动化 entitlement 的配置入口。
- 准备 arm64/x86_64 两种发布配置和架构命名约定。

**L3 测试与兼容（子 Agent 3）**

- 建立 `.runtime/self-tests/phase-00-foundation/` 隔离夹具、Mock API、临时应用数据目录和测试样例。
- 为 `LSMinimumSystemVersion`、Mach-O `minos`、架构切片、签名资源和 WebView 构建目标准备验证脚本/测试。

## 15. 阶段失败与恢复规则

- 阶段门失败：保持当前阶段为 `blocked-by-gate`，不得启动下一阶段；L4 记录首个阻断原因和复测条件。
- 核心契约失败：暂停所有依赖该契约的并行线，只允许 L1/L4 修复并重新跑受影响测试。
- UI/Tauri 测试失败：关闭残留进程，确认使用当前源码和隔离数据后重测，不接受旧安装包或旧内存进程结果。
- 数据、Key、TM 或诊断测试污染真实用户目录：立即停止该测试，删除隔离产物，补充隔离前置检查后重跑。
- macOS 12 二进制或架构门失败：禁止通过修改声明值绕过；必须重新构建依赖或降低兼容依赖版本，并重新扫描整个 `.app`。
- 真实服务 Key、Windows 新版和第三方 Office 缺失：按暂缓/外部验收记录处理，不把缺失条件改写为本地测试通过。

### 15.1 可审计回滚

- 每个阶段完成后由 L4 生成验收记录并保留合并提交 SHA；阶段通过后才允许建立阶段候选标记，失败时使用可审计的 revert，不使用破坏用户改动的 reset/checkout。
- 未发布的阶段候选产物直接作废；正式 Release 只有两个 macOS 资产和全部校验通过后才原子创建，失败时不发布半成品。
- 新基线 schema 如需调整，先在隔离目录生成快照并事务化升级；失败恢复到阶段前快照，不做静默删除或跨旧版本清理。
- 活动任务中的配置、Key 和输出均遵守冻结快照；应用/sidecar 中断按最终决策标为应用中断，不自动伪造可继续状态。
